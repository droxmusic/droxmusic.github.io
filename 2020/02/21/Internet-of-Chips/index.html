Internet of Chips

Derrick Miller
2020-02-21 (Updated: 2020-02-21)

Aren’t Field Programmable Gate Arrays fun?

They have become very popular lately with hardware hackers and makers. Why?

Falling Costs. Open Source Tooling and Hardware. Flexibility. Low Power. Timing.

With these features, they can fill in a role with the fuzzy tag that is “Internet of Things.”

Embedded systems that need both high performance and low power often have:

  • Some System-on-Chip (RPi Compute Module, x86 SMARC, etc).
  • Micro-controllers (AVRs, STM32s, SAMs) as co-processors.

These micro-controllers perform whatever application task but, importantly, can control power or sleep state to the SoC.

So example, a weather platform can have the micro wake the SoC daily from a real-time-clock for all daily tasks.

So long as the current trends continue, we will begin to start seeing more IoT projects with FPGAs as a primary co-processor.

Enter icecored

So as of now I have a USB connected FPGA on an internet connected Single Board Computer that:

  • Provides a bitstream synthesis (compiling for FPGAs) service. Just curl out a zip with a name matching the top level file/module and it will store the resulting .bin in persistent storage.
  • Provides a bitstream storage service. Build offline and upload directly if you’d like.
  • Allows for programming the FPGA by selecting a stored bitstream.
  • Exposes all of this in a semi-sensible API.
  • Does it all in a Docker container.

For my hardware platform I used:

  • The BalenaFin: A Raspberry Pi based platform that provides a few extra fun features such as a 4 pin header for USB.
  • Tinyfpga BX: This has open source tooling (yosys), I also enjoy the very small form-factor.

Also soldered up some headers and pins to a USB breakout board which lets me socket the tinyfpga into the 4 pin USB header.

I made use of the entire Balena stack and prototyped this all out in python.

Here it is: icecored github

Future Work

So right out of the gate: This is not remotely production ready. Things like inputs are not remotely sanitized, and with how loosely I used Invoke, there is probably a bash injection somewhere in here. This was just a small exploration of automating remote FPGA work with this kind of platform. Doing this I have found a few areas of pain, some places to be aware of, and some possible fun applications:

  • Currently, Icecored runs a USB Hub Controller to toggle USB power to reset the FPGA. Unfortunately, the mainline RPi USB Hub for versions before 4 is ganged power control which means a power reset will take out the entire USB hub on the device. This can happen on other SBCs as well. The universal fix to this would be to wire up some GPIO to the reset switch on the FPGA. This software reset wouldn’t take out the entire USB hub then. That or add an extra hub with software power support.
  • It would be nice if the FPGA functionality didn’t have to live in a container. The functionality seems generically useful to any potential container. I would love to integrate a more robust version of this as part of a Host Operating System and have an API available to containers for this functionality. This might be able to make consideration for the former issue since it would be on the hardware specific OS level.
  • A fun possibility I see is in things like retro game emulation. FPGAs have a bit of traction in the retro community as things like edge behaviors are re-implementated in hardware for a more accurate experience. I can envision having a centrally managed fleet of arcade machines.

All that said, it is nice to know that this wasn’t as much of a pain as I thought it was going to be. This could turn into a nice target for other FPGA projects I do.
